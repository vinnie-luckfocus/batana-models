## 1. 项目概述

### 1.1 背景
本项目旨在通过计算机视觉和姿态估计技术，对棒球挥棒动作进行精准分析。通过训练专用的 2D 和 3D Pose 模型，实现对棒球挥棒动作的连续追踪与三维重建，为后续的动作评估、训练反馈和生物力学分析提供数据基础。

### 1.2 目标
- 构建能够精准识别棒球挥棒动作的 **2D Pose 模型**
- 将 2D 动作数据升维为 **3D Pose 模型**，实现三维空间中的动作重建
- 通过半自动数据清洗与人工校验确保输出轨迹的连贯性和准确性
- 利用清洗后的高质量数据优化模型，提升模型在棒球挥棒场景下的表现

### 1.3 数据资源
- **训练视频集**：包含标准、完整的棒球挥棒打击动作视频
  - **规模**：预计 50 ~ 100 段视频（每段 3 ~ 5 秒，90 ~ 150 帧，30 fps）
  - **拍摄要求**：单目固定机位，侧面 90° 视角为首选（可清晰观察挥棒平面）；分辨率不低于 1920×1080；光照均匀，背景简洁
- **验证视频集**：存放于独立目录，用于模型训练后的效果验证
  - **规模**：预计 10 ~ 20 段视频，拍摄标准与训练集一致
- **关键说明**：所有视频中的人物需已签署肖像及数据使用同意书，面部信息可在预处理阶段做模糊处理。

## 2. 技术选型

### 2.1 2D Pose 模型
**主选方案**：`RTMPose-m`（MMPose 项目）
- 兼顾精度与速度，在 COCO 人体关键点检测上表现优异
- 支持视频级时序后处理（如 `RTMPose-t` 或配合 `deciwatch` 做插帧）
- 模型权重需从官方渠道下载，校验 SHA-256

**备选方案**：`YOLOv8-Pose`（ultralytics）—— 若部署端对推理速度要求更高时启用。

### 2.2 3D Pose 模型
**主选方案**：`MotionBERT`
- 基于 Transformer 的时序建模能力强，输出时序稳定性高
- 支持从 2D 关键点序列直接 lifting 到 3D
- 需要约 16 GB 显存进行微调

**备选方案**：`VideoPose3D`—— 若训练数据量不足（<30 段视频）或显存受限时，使用其轻量化版本（如 `VideoPose3D-243f`）。

### 2.3 辅助技术
- **时序滤波**：对 2D 输出使用卡尔曼滤波（Kalman Filter）或光流辅助追踪（Optical Flow），缓解高速挥棒带来的跳变
- **3D 约束优化**：引入 SMPL 人体先验或骨骼长度约束进行 Test-Time Adaptation（TTA），减少手动校正工作量

## 3. 功能需求

### 3.1 2D Pose 动作识别与微调

#### 3.1.1 动作识别 (a)
- 使用 `RTMPose-m` 对训练视频进行逐帧分析。
- 输出每一帧中人体关键点的 2D 坐标。**关键点定义采用扩展 COCO-17 + 2 个球棒关键点 = 19 关键点**，具体定义如下：

| 编号 | 关键点名称 | 说明 |
|------|-----------|------|
| 0 ~ 16 | COCO-17 标准人体关键点 | 鼻子、双眼、双耳、双肩、双肘、双手腕、双臀、双膝、双踝 |
| 17 | `bat_knob` | 球棒握把末端 |
| 18 | `bat_barrel` | 球棒桶部末端 |

- 输出格式为 JSON，单文件结构：
  ```json
  {
    "video_name": "train_001.mp4",
    "fps": 30,
    "resolution": [1920, 1080],
    "keypoint_schema": "coco-17-plus-bat",
    "frames": [
      {
        "frame_id": 0,
        "keypoints": [
          {"id": 0, "x": 320.5, "y": 240.0, "confidence": 0.98},
          ...
        ]
      }
    ]
  }
  ```

#### 3.1.2 轨迹点分析 (b)
- 对识别出的关键点轨迹进行时序分析，识别偏离正常挥棒轨迹的帧。
- 重点关注高动态区域：双手握棒位置（wrist + bat_knob）、肩部旋转、髋部扭转。
- **异常检测算法**（半自动）：
  - 计算相邻帧同一点位的欧氏位移 `d_t = ||P_t - P_{t-1}||`
  - 若 `d_t > μ + 3σ`（其中 `μ`、`σ` 为该点位全序列位移的均值与标准差），则标记为候选跳变帧
  - 对 bat_knob / bat_barrel 增加光流一致性校验：若光流预测位置与实际检测位置偏差 >20 像素，标记为异常

#### 3.1.3 跳变点修正 (c)
- 对候选跳变帧采用**三层修正策略**：
  1. **自动插值**：对单帧跳变，使用前后各 2 帧的 cubic spline 插值修复
  2. **半自动标记**：对连续多帧跳变（>3 帧），在 2D 可视化工具中高亮显示，由标注员确认或重标
  3. **批量复核**：每段视频清洗后，由第二标注员抽查 10% 帧，确认无系统性偏差
- 记录所有修正操作至 `logs/2d_cleaning_log.csv`，字段包含：video_name, frame_id, keypoint_id, original_x, original_y, corrected_x, corrected_y, method, annotator, timestamp
- **人工修正工作量预估**：每段 150 帧视频约需 5 ~ 10 分钟（含自动插值后的人工复核）

#### 3.1.4 反向微调 2D Pose 模型 (d)
- 使用修正后的 2D 轨迹作为 **Ground Truth**，对 `RTMPose-m` 进行有监督微调。
- **微调策略**：
  - 冻结 backbone 前 3 个 stage，开放后续层及 regression head
  - 数据增强：随机旋转（±15°）、亮度变化（±20%）、水平翻转、时序下采样（从 30fps 模拟 15fps）
  - 损失函数：Smooth L1 Loss + 时序一致性损失（相邻帧预测点位的 L1 差值约束）
- **训练参数**：
  - batch size: 16（单卡 16GB 显存）
  - learning rate: 5e-4，余弦退火
  - epochs: 50（配合 early stopping，patience=10）

#### 3.1.5 验证集测试 (e)
- 使用验证目录下的视频对微调后的 2D Pose 模型进行推理测试。
- **评估指标与公式**：

  | 指标 | 公式 / 定义 | 目标值 |
  |------|------------|--------|
  | **时序连贯性（Temporal Consistency, TC）** | `TC = 1 / (1 + mean(||P_t - 2P_{t-1} + P_{t-2}||))`，即二阶差分（加速度）的倒数 | TC > 0.85 |
  | **关键点稳定性（Keypoint Stability, KS）** | `KS = 1 - (跳变帧数 / 总帧数)` | KS > 95% |
  | **漏检率（Missing Rate）** | `confidence < 0.3 的关键点比例` | < 3% |

- 输出验证报告 `reports/2d_validation_report.md`，若未达标返回步骤 3.1.3 继续清洗。

### 3.2 3D Pose 模型转换与优化

#### 3.2.1 2D 转 3D Pose (a)
- 基于 3.1.4 微调后的 2D Pose 结果，使用 `MotionBERT` 将 2D 坐标序列 lifting 为 3D 坐标。
- **前提假设**：当前数据集为**单目视频**，采用单目 lifting 方案。通过在训练过程中注入骨骼长度先验来缓解深度歧义。
- 输出格式：每帧 19 个关键点的 3D 坐标 `(x, y, z)`，单位统一为**厘米（cm）**，以髋部中点（COCO-11 与 12 的中点）为坐标原点。

#### 3.2.2 逐帧异常检测 (b)
- 逐帧读取 3D 动作点序列，检测以下异常：
  - **骨骼长度跳变**：计算每帧中预定义骨骼对的欧氏长度，若某帧长度相对中位数偏差 > 5%，标记为异常
  - **不合理关节角度**：肘部角度 < 30° 或 > 180°；膝部角度 < 10° 或 > 180°
  - **空间位置跳变**：根节点（髋部中点）在相邻帧的 3D 位移 > 15 cm（假设 30fps，对应不可能的人身移动速度）
  - **地面穿透**：任何脚踝或膝盖关键点的 `y` 坐标低于地面平面（`y < 0`，地面设为 `y=0`）

#### 3.2.3 手动校正 (c)
- 对异常帧，使用基于 `Open3D` 或 `matplotlib` 的 3D 骨架播放器进行可视化与手动拖拽校正。
- **校正原则**：
  - 保持骨骼长度在时序上的标准差 < 2%（相对于该骨骼全序列中位数长度）
  - 确保挥棒动作符合人体工学和棒球发力链（Kinetic Chain）规律
  - 头部、躯干、髋部、手臂、球棒的相对位置合理
- **3D 修正策略调整**：
  - 由于单目 lifting 结果中深度轴（z 轴）歧义最大，优先校验并修正 z 轴方向上的不合理位移
  - 对连续多帧异常，采用 SMPL 拟合作为先验：将 2D 检测点投影到 SMPL 模型，用优化算法（如 Levenberg-Marquardt）求解最接近的 3D 姿态，作为手动校正的初始化参考
- **人工修正工作量预估**：每段 150 帧视频约需 15 ~ 25 分钟（含 SMPL 初始化辅助）

#### 3.2.4 3D 模型优化 (d)
- **策略调整**：鉴于 MotionBERT 等 lifter 模型通常需要大规模配对 2D-3D 数据（如 Human3.6M）进行预训练，在项目数据量有限（50~100 段视频）的情况下，**不直接对 MotionBERT 做端到端重训练**，而是采用以下分层优化策略：
  1. **Test-Time Adaptation（TTA）**：在推理阶段，利用骨骼长度损失和时序平滑损失对 MotionBERT 的输出做少量梯度步优化（~10 steps/clip），使结果更贴合当前输入视频
  2. **后处理网络（Post-Refinement Network）**：若数据量达到 80 段以上，训练一个轻量的 2-layer Transformer 后处理网络，以原始 MotionBERT 输出为输入，以人工校正后的 3D 轨迹为标签，学习残差修正
  3. **SMPL 约束模块**：将骨骼长度和关节角度约束编码为可微损失，在 TTA 和后处理网络训练时联合优化

### 3.3 结果验证

#### 3.3.1 验证流程
- 使用验证目录下的独立视频，分别输入到优化后的 **2D Pose 模型**和 **3D Pose  pipeline**中。
- 对验证集的输出结果进行定量和定性评估。

#### 3.3.2 评估指标与公式

**2D 输出评估：**

| 指标 | 公式 / 定义 | 目标值 |
|------|------------|--------|
| 时序平滑度（TC） | `1 / (1 + mean(||P_t - 2P_{t-1} + P_{t-2}||))` | > 0.85 |
| 关键点稳定性（KS） | `1 - (跳变帧数 / 总帧数)` | > 95% |
| 挥棒动作完整性（Swing Completeness, SC） |  SSP 算法计算挥棒起始帧（load）→ 触球帧（contact）→ 随挥结束帧（follow-through）的检测成功率 | > 90% |

**3D 输出评估：**

| 指标 | 公式 / 定义 | 目标值 |
|------|------------|--------|
| 3D 轨迹连贯性（3D-TC） | `1 / (1 + mean(||Q_t - 2Q_{t-1} + Q_{t-2}||))`，Q 为 3D 坐标 | > 0.80 |
| 骨骼长度一致性（BLC） | `mean(σ_bone / μ_bone) * 100%`，即各骨骼长度标准差除以中位数的百分比均值 | < 3% |
| 物理合理性（Physical Plausibility, PP） | 检测到的不合理关节角度帧比例 | < 5% |
| 姿态相似度（Pose Similarity, PS）| 验证集与训练集标准挥棒模板之间的 Dynamic Time Warping (DTW) 平均距离 | < 阈值（在 Phase 3 前用 3 段标杆视频标定） |

#### 3.3.3 验收标准
- 2D Pose 输出在验证视频上的跳变帧比例 **< 5%**
- 3D Pose 输出的骨骼长度时序变化率 **BLC < 3%**
- 验证集上的整体动作序列中，需要手动校正的异常帧比例 **< 10%**
- 输出节奏与实际挥棒节奏一致（2D/3D 的 apex frame 偏移量 < 2 帧）

## 4. 非功能需求

### 4.1 性能要求
- **目标硬件**：NVIDIA GPU with ≥ 16 GB VRAM（如 RTX 4090 / A4000 / V100）
- 单帧 2D Pose 推理耗时 **< 50 ms**（RTX 4090）
- 单段视频（约 3-5 秒，90-150 帧）的全流程处理（2D + 3D TTA）**< 2 分钟**（RTX 4090）
- **CPU Fallback**：若部署环境无 GPU，切换至 `RTMPose-t` + 关闭 TTA，单帧推理目标 < 200 ms

### 4.2 数据管理
- 训练数据、中间结果、模型权重、验证报告需按规范目录存放（见 6.3）
- 所有人工修正的数据需保留版本记录，支持回溯
- 原始视频与 Ground Truth 数据采用只读权限存储，防止误删

### 4.3 安全与隐私
- **数据隐私**：
  - 所有原始视频需在预处理阶段进行面部模糊化（可选）或裁剪
  - 训练与验证数据仅存储于项目内部服务器/加密硬盘中，禁止上传至公共云盘
  - 数据保留期限为项目结束后 2 年，期满后安全删除
- **模型安全**：
  - 预训练模型权重下载后必须校验 SHA-256 哈希值
  - 依赖库版本需锁定（`requirements.txt` 或 `environment.yml`），禁止直接使用未经验证的第三方模型文件
- **部署安全**（若后续部署为服务）：
  - 上传接口需限制文件大小与格式（仅接受 MP4/MOV）
  - 增加 API Key 鉴权与速率限制（Rate Limiting）

### 4.4 可扩展性
- 框架设计需预留接口，未来可扩展至其他棒球动作（投球、接球、跑垒）或其他运动项目
- 关键点 schema 和骨骼定义采用配置文件管理，避免硬编码

### 4.5 可视化
- 提供 2D 和 3D 结果的可视化回放工具
- 支持叠加原始视频与 Pose 骨架
- 支持慢速播放（0.25x / 0.5x）、逐帧步进、关键帧标记（load / contact / follow-through）

## 7. 风险与依赖

### 7.1 技术风险
- **2D 跳变严重**：高速挥棒伴随运动模糊，可能导致 2D 检测在 apex frame（触球瞬间）大面积失效。**缓解**：配合卡尔曼滤波 + 光流追踪，并对 apex 区域进行数据增强训练
- **3D 重建歧义**：单目视频升维 3D 存在深度歧义。**缓解**：引入 SMPL 人体先验和骨骼长度约束；若条件允许，优先收集少量多视角数据用于 3D 标定
- **数据量不足**：若实际可用视频 < 30 段，2D 微调易过拟合，3D 后处理网络难以训练。**缓解**：降低模型容量（RTMPose-t / VideoPose3D-243f）；增加镜像、速度变化、背景替换等数据增强；延长数据收集周期
- **人工校正瓶颈**：若 3D 异常率 > 20%，标注工时将大幅超支。**缓解**：提升 2D 清洗质量（2D 错误会放大到 3D）；使用 SMPL 初始化减少纯手动的修正量

### 7.2 依赖项
- 训练视频和验证视频的获取与标注质量
- 目标部署硬件环境（GPU 型号、显存大小）
- 可视化工具的开发时间
- MMPose / MotionBERT 官方仓库的稳定性与文档支持

### 7.3 缓解方案
- 在数据清洗阶段建立半自动化的异常检测 pipeline
- 考虑引入 SMPL 或棒球专用人体先验模型
- 若视频数量不足，结合时序插值、镜像翻转、速度变化等数据增强手段扩充数据集
- 在 Phase 1 结束后进行一次中期评审（Mid-term Review），评估 2D 清洗效果和实际视频数量，必要时调整后续技术方案

## 8. 附录

### 8.1 关键术语
- **Pose Estimation**：姿态估计，从图像/视频中提取人体关键点位置的技术
- **Fine-tuning**：微调，在预训练模型的基础上，使用领域特定数据继续训练
- **Lifting**：升维，将 2D 坐标映射到 3D 空间的过程
- **Kinetic Chain**：发力链，棒球挥棒时力量从下肢经躯干传递至上肢和球棒的连贯过程
- **TTA (Test-Time Adaptation)**：测试时自适应，在 inference 阶段针对当前输入做少量优化，以提升输出质量

### 8.2 参考模型与官方链接
- **2D Pose**：

### 8.3 目录结构建议
```
project/
├── data/
│   ├── raw_videos/            # 原始视频（只读）
│   │   ├── train/             # 训练视频
│   │   └── val/               # 验证视频
│   ├── train_2d_keypoints/    # 原始 2D 输出（JSON）
│   ├── train_2d_cleaned/      # 修正后的 2D 数据 + 清洗日志
│   ├── train_3d_keypoints/    # 原始 3D 输出（npz / JSON）
│   └── train_3d_cleaned/      # 修正后的 3D 数据
├── models/
│   ├── 2d_pose/               # 2D Pose 模型权重
│   │   ├── rtmpose_m_coco.pth
│   │   └── rtmpose_m_finetuned.pth
│   └── 3d_pose/               # 3D Pose 模型权重
│       ├── motionbert_pretrained.pth
│       └── motionbert_tta.yaml
├── configs/
│   ├── keypoint_schema.yaml   # 关键点定义与骨骼连接
│   ├── train_2d.yaml
│   └── train_3d.yaml
├── scripts/
│   ├── inference_2d.py
│   ├── inference_3d.py
│   ├── visualize_2d.py
│   ├── visualize_3d.py
│   ├── finetune_2d.py
│   └── post_refine_3d.py
├── logs/                      # 训练日志与数据清洗记录
├── reports/                   # 验证报告
├── requirements.txt
└── README.md
```
