## 1. 项目概述

### 1.1 背景
本项目旨在通过计算机视觉和姿态估计技术，对棒球挥棒动作进行精准分析。通过训练专用的 2D 和 3D Pose 模型，实现对棒球挥棒动作的连续追踪与三维重建，为后续的动作评估、训练反馈和生物力学分析提供数据基础。

### 1.2 目标
- 构建能够精准识别棒球挥棒动作的 **2D Pose 模型**
- 将 2D 动作数据升维为 **3D Pose 模型**，实现三维空间中的动作重建
- **通过算法约束与自动化代理（Agent）确保输出轨迹的连贯性与准确性**
- 利用自动清洗后的高质量数据优化模型，提升模型在棒球挥棒场景下的表现

### 1.3 数据资源
- **训练视频集**：包含标准、完整的棒球挥棒打击动作视频
  - **规模**：预计 50 ~ 100 段视频（每段 3 ~ 5 秒，90 ~ 150 帧，30 fps）
  - **拍摄要求**：单目固定机位，侧面 90° 视角为首选（可清晰观察挥棒平面）；分辨率不低于 1920×1080；光照均匀，背景简洁
- **验证视频集**：存放于独立目录，用于模型训练后的效果验证
  - **规模**：预计 10 ~ 20 段视频，拍摄标准与训练集一致

### 1.4 项目执行模式（零人工干预）
> **本项目的核心约束**：一旦项目启动，发起人将不再参与任何中间决策、数据校验或技术选型讨论。项目中所有需要判断、修正、调优的环节，均由自动化 Agent 独立完成。
>
> **Agent 的职责边界**：
> - 对 2D/3D 输出中的跳变、异常、遮挡进行算法层面的自动修正
> - 对模型训练参数、损失函数权重、数据增强策略进行自主调优
> - 在遇到技术分歧或方案选择时，Agent 依据预设的生物力学约束与数学平滑性原则自行裁决
> - 项目最终只向发起人交付可运行的模型权重、推理脚本、验证报告和说明文档
>
> **基本原则**：棒球挥棒是一个连续不断的刚体运动，即使存在短暂遮挡，关键点的物理位置也不可能发生瞬时跳变。任何被检测到的跳变在逻辑上即为错误，Agent 必须将其修正为符合运动学连续性的轨迹。

## 2. 技术选型

### 2.1 2D Pose 模型
**主选方案**：`RTMPose-m`（MMPose 项目）
- 兼顾精度与速度，在 COCO 人体关键点检测上表现优异
- 支持视频级时序后处理（如 `RTMPose-t` 或配合 `deciwatch` 做插帧）
- 模型权重需从官方渠道下载，校验 SHA-256

**备选方案**：`YOLOv8-Pose`（ultralytics）—— 若部署端对推理速度要求更高时启用。**由 Agent 依据验证集指标自行决定切换时机。**

### 2.2 3D Pose 模型
**主选方案**：`MotionBERT`
- 基于 Transformer 的时序建模能力强，输出时序稳定性高
- 支持从 2D 关键点序列直接 lifting 到 3D
- 需要约 16 GB 显存进行微调

**备选方案**：`VideoPose3D`—— 若训练数据量不足（<30 段视频）或显存受限时，由 Agent 自动降级至轻量化版本（如 `VideoPose3D-243f`）。

### 2.3 自动化修正 Agent 架构
本项目配置一个核心代理：**`Trajectory Refinement Agent`**

- **输入**：模型原始输出的 2D/3D 关键点序列
- **核心能力**：
  1. **异常检测**：基于时序连续性、光流一致性、骨骼长度约束检测跳变帧
  2. **自动修正**：综合运用样条插值、卡尔曼滤波、SMPL 拟合、逆运动学（IK）优化等手段修复异常
  3. **迭代反馈**：将修正后的高质量数据作为新 Ground Truth 反向注入模型训练流程
- **决策原则**：
  - 物理连续性优先：轨迹必须满足时间上的二阶光滑（加速度有界）
  - 骨骼刚性优先：人体骨骼长度在时序上基本恒定（允许 <3% 的合理形变）
  - 遮挡可推断优先：被遮挡点应依据前后帧和运动学规律进行推断补全，而非保留跳变

## 3. 功能需求

### 3.1 2D Pose 动作识别与微调

#### 3.1.1 动作识别 (a)
- 使用 `RTMPose-m` 对训练视频进行逐帧分析。
- 输出每一帧中人体关键点的 2D 坐标。**关键点定义采用扩展 COCO-17 + 2 个球棒关键点 = 19 关键点**，具体定义如下：

| 编号 | 关键点名称 | 说明 |
|------|-----------|------|
| 0 ~ 16 | COCO-17 标准人体关键点 | 鼻子、双眼、双耳、双肩、双肘、双手腕、双臀、双膝、双踝 |
| 17 | `bat_knob` | 球棒握把末端 |
| 18 | `bat_barrel` | 球棒桶部末端 |

- 输出格式为 JSON，单文件结构：
  ```json
  {
    "video_name": "train_001.mp4",
    "fps": 30,
    "resolution": [1920, 1080],
    "keypoint_schema": "coco-17-plus-bat",
    "frames": [
      {
        "frame_id": 0,
        "keypoints": [
          {"id": 0, "x": 320.5, "y": 240.0, "confidence": 0.98},
          ...
        ]
      }
    ]
  }
  ```

#### 3.1.2 轨迹点分析 (b)
- 对识别出的关键点轨迹进行时序分析，识别偏离正常挥棒轨迹的帧。
- 重点关注高动态区域：双手握棒位置（wrist + bat_knob）、肩部旋转、髋部扭转。
- **异常检测算法**（全自动，由 Agent 执行）：
  - 计算相邻帧同一点位的欧氏位移 `d_t = ||P_t - P_{t-1}||`
  - 若 `d_t > μ + 3σ`（其中 `μ`、`σ` 为该点位全序列位移的均值与标准差），则标记为候选跳变帧
  - 对 bat_knob / bat_barrel 增加光流一致性校验：若光流预测位置与实际检测位置偏差 >20 像素，标记为异常
  - **Agent 自主裁决**：若某区域连续高动态导致统计学阈值失效，Agent 可切换至基于相邻帧加速度阈值（`||P_t - 2P_{t-1} + P_{t-2}|| > 50 px`）进行二次判定

#### 3.1.3 跳变点自动修正 (c)
> **完全由 Trajectory Refinement Agent 自动执行，无需人类确认。**

- 对候选跳变帧采用**全自动修正策略**：
  1. **单帧插值**：对孤立跳变帧（前后各 1 帧正常），使用 cubic spline 插值修复
  2. **多帧序列修复**：对连续 2~5 帧跳变，使用 **Kalman Smoother** 或 **Rauch-Tung-Striebel (RTS)** 平滑滤波，基于运动学模型进行最优估计
  3. **长段遮挡推断**：对连续 >5 帧的异常或低置信度区域（如球棒高速挥动导致的严重模糊），使用 **光流追踪 + 刚性棒模型约束** 进行轨迹推算：
     - 将球棒视为刚性杆，`bat_knob` 与 `bat_barrel` 间距必须在全序列中保持恒定（±5%）
     - 双手腕与 `bat_knob` 的相对位置关系由邻近正常帧的统计分布推断
  4. **全局二次优化**：对整条视频的 2D 轨迹施加 **时序全变分最小化（Temporal Total Variation Minimization）**，确保二阶导数（加速度）在全序列上能量最小

- Agent 记录所有修正操作至 `logs/2d_cleaning_log.csv`，字段包含：video_name, frame_id, keypoint_id, original_x, original_y, corrected_x, corrected_y, method, confidence_score, timestamp
- **修正后必须通过自动质检**：修正后的序列 TC 值必须 ≥ 0.85，否则 Agent 自动回溯并尝试更高阶的平滑策略

#### 3.1.4 反向微调 2D Pose 模型 (d)
- 使用 Agent 修正后的 2D 轨迹作为 **Ground Truth**，对 `RTMPose-m` 进行有监督微调。
- **微调策略**：
  - 冻结 backbone 前 3 个 stage，开放后续层及 regression head
  - 数据增强：随机旋转（±15°）、亮度变化（±20%）、水平翻转、时序下采样（从 30fps 模拟 15fps）
  - 损失函数：Smooth L1 Loss + 时序一致性损失（相邻帧预测点位的 L1 差值约束）
- **训练参数**（Agent 可在训练过程中根据 loss 曲线自主调整 LR 和 Epoch）：
  - batch size: 16（单卡 16GB 显存）
  - 初始 learning rate: 5e-4，余弦退火
  - max epochs: 50（配合 early stopping，patience=10，由 Agent 监控并触发）

#### 3.1.5 验证集测试 (e)
- 使用验证目录下的视频对微调后的 2D Pose 模型进行推理测试。
- **评估指标与公式**：

  | 指标 | 公式 / 定义 | 目标值 |
  |------|------------|--------|
  | **时序连贯性（Temporal Consistency, TC）** | `TC = 1 / (1 + mean(||P_t - 2P_{t-1} + P_{t-2}||))` | TC > 0.85 |
  | **关键点稳定性（Keypoint Stability, KS）** | `KS = 1 - (跳变帧数 / 总帧数)` | KS > 95% |
  | **漏检率（Missing Rate）** | `confidence < 0.3 的关键点比例` | < 3% |

- 输出验证报告 `reports/2d_validation_report.md`。**若未达标，Agent 自动返回步骤 3.1.3 调整修正策略或数据增强参数，无需人类介入。**

### 3.2 3D Pose 模型转换与优化

#### 3.2.1 2D 转 3D Pose (a)
- 基于 3.1.4 微调后的 2D Pose 结果，使用 `MotionBERT` 将 2D 坐标序列 lifting 为 3D 坐标。
- **前提假设**：当前数据集为**单目视频**，采用单目 lifting 方案。通过在训练过程中注入骨骼长度先验来缓解深度歧义。
- 输出格式：每帧 19 个关键点的 3D 坐标 `(x, y, z)`，单位统一为**厘米（cm）**，以髋部中点（COCO-11 与 12 的中点）为坐标原点。

#### 3.2.2 逐帧异常检测 (b)
- 逐帧读取 3D 动作点序列，检测以下异常：
  - **骨骼长度跳变**：计算每帧中预定义骨骼对的欧氏长度，若某帧长度相对中位数偏差 > 5%，标记为异常
  - **不合理关节角度**：肘部角度 < 30° 或 > 180°；膝部角度 < 10° 或 > 180°
  - **空间位置跳变**：根节点（髋部中点）在相邻帧的 3D 位移 > 15 cm（假设 30fps，对应不可能的人身移动速度）
  - **地面穿透**：任何脚踝或膝盖关键点的 `y` 坐标低于地面平面（`y < 0`，地面设为 `y=0`）

#### 3.2.3 3D 轨迹自动修正 (c)
> **完全由 Trajectory Refinement Agent 自动执行，无需人类确认。**

- 对 3D 异常帧，Agent 采用以下**全自动约束优化策略**：
  1. **SMPL 先验拟合**：将 2D 检测点投影到 SMPL 模型，使用 **Levenberg-Marquardt 优化**求解满足 2D 重投影误差最小且骨骼长度最合理的 3D 姿态，作为修正的初始解
  2. **逆运动学（IK）微调**：对球棒与上肢的相对位置，使用基于 Jacobian 的 IK 求解器，在保持 `bat_knob` 与双手腕约束关系的前提下，将球棒调整至最符合运动学规律的位置
  3. **时序光滑投影**：将单帧修正结果放入一个 **滑动窗口优化器**（Sliding Window Optimizer，窗口大小 7 帧），最小化窗口内所有帧的骨骼长度方差 + 关节加速度 + 地面穿透惩罚项
  4. **全局 Bundle Adjustment**：对整条 3D 序列，Agent 运行一次轻量级的全局优化，目标函数为：
     ```
     L_total = λ_1 * L_reproj + λ_2 * L_bone_length + λ_3 * L_temporal_smooth + λ_4 * L_ground_penetration
     ```
     其中各 λ 权重由 Agent 依据异常类型动态调整（如骨骼长度异常激增时提高 λ_2）

- **修正原则**（硬编码进 Agent 决策逻辑）：
  - 保持骨骼长度在时序上的标准差 < 2%（相对于该骨骼全序列中位数长度）
  - 确保挥棒动作符合人体工学和棒球发力链（Kinetic Chain）规律
  - 头部、躯干、髋部、手臂、球棒的相对位置合理
  - 单目 lifting 中深度轴（z 轴）歧义最大，Agent 优先校验 z 轴方向上的不合理位移

- Agent 记录所有修正操作至 `logs/3d_cleaning_log.csv`。

#### 3.2.4 3D 模型优化 (d)
- **策略**：鉴于 MotionBERT 等 lifter 模型通常需要大规模配对 2D-3D 数据（如 Human3.6M）进行预训练，在项目数据量有限（50~100 段视频）的情况下，**不直接对 MotionBERT 做端到端重训练**，而是采用以下分层优化策略（由 Agent 自动执行）：
  1. **Test-Time Adaptation（TTA）**：在推理阶段，利用骨骼长度损失和时序平滑损失对 MotionBERT 的输出做少量梯度步优化（~10 steps/clip），使结果更贴合当前输入视频
  2. **后处理网络（Post-Refinement Network）**：若数据量达到 80 段以上，Agent 自动训练一个轻量的 2-layer Transformer 后处理网络，以原始 MotionBERT 输出为输入，以 Agent 修正后的 3D 轨迹为标签，学习残差修正
  3. **SMPL 约束模块**：将骨骼长度和关节角度约束编码为可微损失，在 TTA 和后处理网络训练时联合优化

### 3.3 结果验证

#### 3.3.1 验证流程
- 使用验证目录下的独立视频，分别输入到优化后的 **2D Pose 模型**和 **3D Pose  pipeline**中。
- 对验证集的输出结果进行定量和定性评估。**验证报告由 Agent 自动生成并写入 `reports/`。**

#### 3.3.2 评估指标与公式

**2D 输出评估：**

| 指标 | 公式 / 定义 | 目标值 |
|------|------------|--------|
| 时序平滑度（TC） | `1 / (1 + mean(||P_t - 2P_{t-1} + P_{t-2}||))` | > 0.85 |
| 关键点稳定性（KS） | `1 - (跳变帧数 / 总帧数)` | > 95% |
| 挥棒动作完整性（Swing Completeness, SC） |  SSP 算法计算挥棒起始帧（load）→ 触球帧（contact）→ 随挥结束帧（follow-through）的检测成功率 | > 90% |

**3D 输出评估：**

| 指标 | 公式 / 定义 | 目标值 |
|------|------------|--------|
| 3D 轨迹连贯性（3D-TC） | `1 / (1 + mean(||Q_t - 2Q_{t-1} + Q_{t-2}||))`，Q 为 3D 坐标 | > 0.80 |
| 骨骼长度一致性（BLC） | `mean(σ_bone / μ_bone) * 100%`，即各骨骼长度标准差除以中位数的百分比均值 | < 3% |
| 物理合理性（Physical Plausibility, PP） | 检测到的不合理关节角度帧比例 | < 5% |
| 姿态相似度（Pose Similarity, PS）| 验证集与训练集标准挥棒模板之间的 Dynamic Time Warping (DTW) 平均距离 | < 阈值（在 Phase 3 前由 Agent 用 3 段标杆视频自动标定） |

#### 3.3.3 验收标准
- 2D Pose 输出在验证视频上的跳变帧比例 **< 5%**
- 3D Pose 输出的骨骼长度时序变化率 **BLC < 3%**
- 验证集上的整体动作序列中，**Agent 自动修正后无需人类复核即可达到可直接使用的质量标准**
- 输出节奏与实际挥棒节奏一致（2D/3D 的 apex frame 偏移量 < 2 帧）

## 4. 非功能需求

### 4.1 性能要求
- **目标硬件**：NVIDIA GPU with ≥ 16 GB VRAM（如 RTX 4090 / A4000 / V100）
- 单帧 2D Pose 推理耗时 **< 50 ms**（RTX 4090）
- 单段视频（约 3-5 秒，90-150 帧）的全流程处理（2D + 3D TTA）**< 2 分钟**（RTX 4090）
- **CPU Fallback**：若部署环境无 GPU，由 Agent 自动切换至 `RTMPose-t` + 关闭 TTA，单帧推理目标 < 200 ms

### 4.2 数据管理
- 训练数据、中间结果、模型权重、验证报告需按规范目录存放（见 6.3）
- 所有自动修正的数据需保留版本记录与算法参数快照，支持回溯
- 原始视频与 Ground Truth 数据采用只读权限存储，防止误删

### 4.3 安全与隐私
- **数据隐私与伦理**：
  - 视频数据为内部训练资料，项目启动前需确保已具备合法使用权
  - 预处理阶段将自动对人物面部区域进行模糊化（由脚本自动处理，无需人工审查）
  - 数据仅存储于本地环境，禁止上传至公共云盘或第三方平台
- **模型安全**：
  - 预训练模型权重下载后必须校验 SHA-256 哈希值
  - 依赖库版本需锁定（`requirements.txt` 或 `environment.yml`），禁止直接使用未经验证的第三方模型文件
- **部署安全**（若后续部署为服务）：
  - 上传接口需限制文件大小与格式（仅接受 MP4/MOV）
  - 增加 API Key 鉴权与速率限制（Rate Limiting）

### 4.4 可扩展性
- 框架设计需预留接口，未来可扩展至其他棒球动作（投球、接球、跑垒）或其他运动项目
- 关键点 schema 和骨骼定义采用配置文件管理，避免硬编码

### 4.5 可视化
- 提供 2D 和 3D 结果的可视化回放工具（用于 Agent 内部质检与最终交付展示）
- 支持叠加原始视频与 Pose 骨架
- 支持慢速播放（0.25x / 0.5x）、逐帧步进、关键帧标记（load / contact / follow-through）

## 7. 风险与依赖

### 7.1 技术风险
- **2D 跳变严重**：高速挥棒伴随运动模糊，可能导致 2D 检测在 apex frame（触球瞬间）大面积失效。**缓解**：配合卡尔曼滤波 + 光流追踪 + 刚性棒模型约束；Agent 会迭代尝试多种插值策略直至 TC 达标
- **3D 重建歧义**：单目视频升维 3D 存在深度歧义。**缓解**：引入 SMPL 人体先验、骨骼长度约束和滑动窗口优化器；若条件允许，Agent 可自动收集并分析少量多视角数据用于辅助标定
- **数据量不足**：若实际可用视频 < 30 段，2D 微调易过拟合，3D 后处理网络难以训练。**缓解**：Agent 自动降级模型容量（RTMPose-t / VideoPose3D-243f）；增加镜像、速度变化、背景替换等数据增强；延长数据收集周期（Agent 自行判断是否暂停并发出提示）
- **Agent 自动修正失败**：若某段视频存在持续严重遮挡或画质过低，算法修正可能无法达到验收标准。**缓解**：将该段视频标记为 `UNREFINABLE` 并从训练集中剔除；Agent 记录原因并汇总为 `reports/low_quality_videos.md`，不阻塞整体 pipeline

### 7.2 依赖项
- 训练视频和验证视频的获取质量与分辨率
- 目标部署硬件环境（GPU 型号、显存大小）
- MMPose / MotionBERT / SMPL 官方仓库的稳定性与文档支持
- 项目启动后发起人不再干预，Agent 必须具备完整的异常处理与自主降级能力

### 7.3 缓解方案
- 在数据清洗阶段建立全自动化的异常检测与修正 pipeline
- 考虑引入 SMPL 或棒球专用人体先验模型
- 若视频数量不足，结合时序插值、镜像翻转、速度变化等数据增强手段扩充数据集
- 在 Phase 2 结束后由 Agent 进行一次自动中期评审（Auto Mid-term Review），基于 2D 验证指标和实际视频数量，自主决定是否需要调整后续技术方案

## 8. 附录

### 8.1 关键术语
- **Pose Estimation**：姿态估计，从图像/视频中提取人体关键点位置的技术
- **Fine-tuning**：微调，在预训练模型的基础上，使用领域特定数据继续训练
- **Lifting**：升维，将 2D 坐标映射到 3D 空间的过程
- **Kinetic Chain**：发力链，棒球挥棒时力量从下肢经躯干传递至上肢和球棒的连贯过程
- **TTA (Test-Time Adaptation)**：测试时自适应，在 inference 阶段针对当前输入做少量优化，以提升输出质量

- **2D Pose**：
  - RTMPose: https://github.com/open-mmlab/mmpose

- **3D Pose**：
  - MotionBERT: https://github.com/Walter0807/MotionBERT
  - VideoPose3D: https://github.com/facebookresearch/VideoPose3D
- **人体先验**：
